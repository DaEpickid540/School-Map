The Problem

Mason High School is a large, architecturally complex building — multiple pods, long connecting hallways, irregular room numbering, and three floors that don't align cleanly. Students get lost on their first day. Visitors struggle to find specific rooms. Even staff occasionally have to ask for directions. There was no official digital map, no coordinate data, and no routing system of any kind. I built one from scratch.

Missing Digital Infrastructure

Most modern public buildings have some form of digital wayfinding — an app, a kiosk, or at minimum a floor plan PDF. Mason had none of these for students. The building's complexity made verbal directions unreliable. The absence of a navigation system wasn't just an inconvenience — it was a gap in the school's infrastructure. This project fills that gap.

Why a Coordinate System Was Necessary

You cannot build a routing system without knowing where things are. I had to establish a pixel-accurate coordinate for every room, hub, stairwell, and hallway node across all three floors. I created a coordinate system from scratch using a top-left origin and consistent pixel mapping — then annotated every room using a professional labeling workflow, verifying positions against the actual floor plan images.

The Challenge of Multi-Floor Mapping

Mapping a single floor is straightforward. Mapping three floors that transition through specific stairwells — with each stairwell connecting only certain pods — is a fundamentally different problem. I modeled each stairwell as a named node in the routing graph, explicitly connecting the correct pods on each floor, so paths could flow naturally across floors while remaining accurate to the physical building.

Accuracy and Consistency

Bad coordinate data produces misleading routes — worse than no data at all. I approached accuracy as an engineering requirement, not a nice-to-have. Every room position was read directly from floor plan images at high zoom, cross-referenced against room labels, and structured in a consistent format. The data model is deterministic: same input always produces the same output.

Why Graph-Based Routing Matters

A simple lookup table can tell you where a room is. Only a routing graph can tell you how to get there. I built a graph where every room, pod, stairwell, and hallway connector is a node, and every physical connection is an edge. BFS pathfinding then computes the shortest valid path — handling multi-floor transitions, pod-to-pod hops, and stairwell logic automatically.

Designed to Scale

The system is modular by design. Room-to-pod mappings live in separate files per pod. Coordinates are isolated per floor. The graph is defined once and shared across the routing and display layers. Adding a new room means adding one entry in one file — nothing else changes. The same architecture could be adapted to any other building with minimal effort.

What This Demonstrates

This project is an exercise in engineering thinking applied to a real, unsolved problem. I didn't start with a framework or a tutorial — I started with a building, a problem, and first principles. I identified what data was needed, built a system to capture it accurately, designed a graph model that reflects physical reality, and wrapped it in a UI that's fast and usable. Every decision was made to serve correctness, maintainability, and real-world utility. That's what engineering is.
